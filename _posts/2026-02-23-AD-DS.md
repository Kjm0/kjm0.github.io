## Network Configuration – Static IP Assignment

### Objective
Ensure the Domain Controller maintains a consistent IP address to prevent DNS and authentication failures.

### Why This Matters
Active Directory is tightly integrated with DNS.  
If a Domain Controller receives a new IP address from DHCP:

- Clients may fail authentication
- Domain join operations may fail
- DNS records may become inconsistent
- Replication issues may occur (in multi-DC environments)

In production, Domain Controllers always use static IP addresses.

### Configuration

Configured the Domain Controller with (Using Pfsense, which will be covered later on):

- Static IP: 10.10.10.10 
- Subnet Mask: 255.255.255.0
- Default Gateway: 10.10.10.1
- Preferred DNS: 10.10.10.10 (self)

![IPv4 static configuration window](/assets/img/AD-DS/ipv4-dns-config.png)

### Validation

Executed:

```
ipconfig /all
```

Verified:
- DHCP Enabled: No
- DNS server points to local DC IP

![ipconfig /all showing DHCP disabled](/assets/img/AD-DS/dhcp-no.png)

Tested DNS resolution:

```
nslookup kemocorp.com
```

## Organizational Unit (OU) Design

### Objective
Design an OU structure that supports delegation and Group Policy application.

### OU Structure

```
kemocorp.com
│
├── Employees
├── Workstations
├── Servers
├── Admin Accounts
└── Service Accounts
```

![Organizational Unit (OU) Structure](/assets/img/AD-DS/ou-structure.png)
![Group Policy Object (GPO) Structure](/assets/img/AD-DS/gp-structure.png)


### Design Rationale

- Users separated from computers for targeted GPO application.
- Admin accounts isolated to apply stricter security policies.
- Avoided deep OU nesting to reduce complexity.

### Production Notes

OU structure in production environments is designed around:

- Policy boundaries
- Delegation requirements
- Administrative scope

## User and Group Management

### Objective
Implement proper permission assignment using security groups.

### Security Model Used

Followed the AGDLP model:

Accounts → Global Groups → Domain Local Groups → Permissions

### Implementation

- Created Global Security Groups for departments
- Created Domain Local Groups for resource access
- Assigned permissions to groups, not individual users
  
![] (assets/img/AD-DS/gpo.png)

### Common Help Desk Tasks Demonstrated

- Reset user password
- Unlock locked account
- Disable and enable user
- Add user to group
- Remove user from group

### Production Notes

In production:
- Permissions are never assigned directly to users.
- Group membership changes are logged and often ticket-based.

## PowerShell Administration

Although tasks were initially performed via GUI, equivalent PowerShell commands were tested for automation readiness.

### Example Tasks

Reset password:

```
Set-ADAccountPassword -Identity "insertUserNameHere" -Reset -NewPassword (ConvertTo-SecureString "TempPass123!" -AsPlainText -Force)
```

Force password change at next logon:

```
Set-ADUser -Identity "insertUserNameHere" -ChangePasswordAtLogon $true
```

Add user to group:

```
Add-ADGroupMember -Identity "HR_Group" -Members "insertUserNameHere"
```

List inactive accounts:

```
Search-ADAccount -AccountInactive -TimeSpan 90.00:00:00
```

### Production Insight

In enterprise environments:

- Bulk onboarding and offboarding is scripted.
- Reports are generated via PowerShell.
- Scheduled tasks automate recurring administrative tasks.
- Integration with ServiceNow or similar ITSM tools triggers automation workflows.

GUI is commonly used for single-user tasks.  
PowerShell is used for scale and reporting.
